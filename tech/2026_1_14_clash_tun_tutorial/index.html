Clash/Mihomo TUN 模式完全解析

从 Wireshark 抓包理解透明代理的工作原理

1. 基础概念

1.1 什么是 TUN 模式？

TUN 模式是一种透明代理技术，应用程序无需配置代理，所有流量自动被 Clash 接管。

核心组件：

应用程序 → 虚拟网卡(TUN) → Clash → 物理网卡 → 互联网
           198.18.0.x      处理分流    172.26.x.x

1.2 关键 IP 地址

类型IP 地址说明
物理网卡172.26.x.x真实网络 IP
虚拟网卡198.18.0.1TUN 设备 IP
虚拟 DNS198.18.0.2Clash DNS 服务
Fake-IP 池198.18.0.0/15虚拟 IP 池（131,072 个地址）

2. DNS 劫持机制

2.1 双层劫持

Clash 使用两层机制确保所有 DNS 查询被劫持：

第一层：systemd-resolved 配置

$ resolvectl status
Link 7 (Mihomo):
    DNS Servers: 198.18.0.2
    DNS Domain: ~.  # ← 关键！匹配所有域名，优先级最高

第二层：策略路由拦截

# 即使你指定 DNS 服务器
$ nslookup google.com 8.8.8.8
# 仍然被劫持！

# 原因：
$ ip route get 8.8.8.8
8.8.8.8 via 198.18.0.2 dev Mihomo  # 流量走虚拟网卡

原理：

  1. 策略路由在内核级别拦截所有流量
  2. Clash 检测端口 53（DNS 端口）
  3. 拦截并返回 Fake-IP
  4. 伪装响应源地址（让你以为来自 8.8.8.8）

2.2 DNS Domain: ~. 的作用

systemd-resolved 的优先级：

优先级 1: 精确域名匹配（如 google.com）
优先级 2: 路由域匹配（~.）  ← Clash 在这里
优先级 3: 默认路由（+DefaultRoute）

~. = 匹配所有域名的路由域配置，优先级高于物理网卡的默认 DNS。

3. 策略路由

3.1 为什么流量走虚拟网卡？

你看到的主路由表：

$ ip route show
default via 172.26.255.254 dev wlp0s20f3  # 默认走物理网卡

但 Linux 有多张路由表！

# 策略路由规则
$ ip rule show
9002: not from all iif lo lookup 2022  # ← 优先级高
32766: from all lookup main            # ← 主路由表

# 表 2022 的内容
$ ip route show table 2022
default via 198.18.0.2 dev Mihomo  # ← 默认走虚拟网卡！

工作原理：

  1. 应用发起连接（如访问 Google）
  2. 内核查询策略路由规则（按优先级）
  3. 规则 9002 匹配 → 查询表 2022
  4. 表 2022：所有流量走虚拟网卡
  5. 流量进入 Clash

4. Fake-IP 核心原理

4.1 什么是 Fake-IP？

Fake-IP 是 Clash 为每个域名分配的虚拟 IP 地址。

示例：

$ nslookup google.com 198.18.0.2
Address: 198.18.0.215  # ← Fake-IP

$ nslookup baidu.com 198.18.0.2
Address: 198.18.0.209  # ← 另一个 Fake-IP

4.2 Fake-IP 映射表

Clash 内部维护一个核心映射表：

┌─────────────────────────────────────┐
│  Fake-IP 映射表（核心数据结构）      │
├─────────────────────────────────────┤
│ Fake-IP        ↔  域名               │
├─────────────────────────────────────┤
│ 198.18.0.215  →  google.com         │
│ 198.18.0.209  →  baidu.com          │
│ 198.18.0.71   →  github.com         │
└─────────────────────────────────────┘

特点：

  • 持久化：同一域名在缓存期内返回相同的 Fake-IP
  • 双向查询：
    • DNS 查询时：域名 → Fake-IP（写入）
    • 流量处理时：Fake-IP → 域名（读取）

4.3 为什么需要 Fake-IP？

传统方式的问题：

DNS 返回真实 IP → 需要 iptables 拦截流量 → 复杂

Fake-IP 的优势：

DNS 返回虚拟 IP → 策略路由自动捕获 → 简单
         ↓
    映射表记录
         ↓
    反查域名匹配规则 → 灵活

5. 流量处理流程

5.1 Clash 如何处理 TUN 流量

┌─────────────────────────────────────────────┐
│ ① 从 TUN 读取数据包                         │
│    目标: 198.18.0.215:443                   │
└──────────────┬──────────────────────────────┘
               ↓
┌─────────────────────────────────────────────┐
│ ② 查询 Fake-IP 映射表 ⭐                   │
│    198.18.0.215 → google.com               │
└──────────────┬──────────────────────────────┘
               ↓
┌─────────────────────────────────────────────┐
│ ③ 匹配规则                                  │
│    DOMAIN-KEYWORD,google,Proxy ✅          │
└──────────────┬──────────────────────────────┘
               ↓
┌─────────────────────────────────────────────┐
│ ④ 执行动作                                  │
│    Proxy: 通过代理服务器                    │
│    DIRECT: 查真实 IP 后直连                │
└─────────────────────────────────────────────┘

5.2 规则配置示例

rules:
  # 广告拦截
  - DOMAIN-KEYWORD,ads,REJECT
  
  # 国内直连
  - DOMAIN-SUFFIX,cn,DIRECT
  - GEOIP,CN,DIRECT
  
  # 国外代理
  - DOMAIN-SUFFIX,google.com,Proxy
  - DOMAIN-KEYWORD,youtube,Proxy
  
  # 兜底规则
  - MATCH,Proxy

5.3 代理 vs 直连

代理流量（如 Google）：

浏览器 → Clash(TUN) → Clash 查规则(Proxy)
                    ↓
              连接代理服务器（加密隧道）
                    ↓
              代理服务器访问 Google

直连流量（如百度）：

浏览器 → Clash(TUN) → Clash 查规则(DIRECT)
                    ↓
              查询真实 IP (220.181.38.148)
                    ↓
              物理网卡直连百度

6. 完整工作流程

访问 Google 的完整过程

1. 浏览器输入: google.com
   ↓
2. DNS 查询被劫持
   • 查询发往 198.18.0.2 (Clash DNS)
   • Clash 分配 Fake-IP: 198.18.0.215
   • 写入映射表: google.com ↔ 198.18.0.215
   ↓
3. 浏览器连接 198.18.0.215:443
   ↓
4. 策略路由拦截
   • 查询规则 9002 → 表 2022
   • 流量进入 Mihomo 虚拟网卡
   ↓
5. Clash 处理
   • 读取数据包: 目标 198.18.0.215:443
   • 查映射表: 198.18.0.215 → google.com
   • 匹配规则: DOMAIN-KEYWORD,google,Proxy
   • 选择代理节点
   ↓
6. 建立代理连接
   • 从物理网卡连接代理服务器
   • 通过加密隧道访问 Google
   ↓
7. 数据双向转发
   浏览器 ←→ Clash ←→ 代理 ←→ Google

📊 核心概念总结

Clash TUN 模式的三大支柱

┌─────────────────────────────────────────┐
│ 1. DNS 劫持 (双层机制)                  │
│    • DNS Domain: ~.                     │
│    • 策略路由拦截                        │
│    → 所有 DNS 查询返回 Fake-IP          │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 2. Fake-IP 映射表 (核心数据结构)        │
│    • 连接 DNS 和流量处理                │
│    • 域名 ↔ Fake-IP 双向映射            │
│    → 使基于域名的规则匹配成为可能        │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 3. 策略路由 (内核级拦截)                │
│    • 多路由表机制                        │
│    • 优先级控制                          │
│    → 所有流量强制进入 Clash              │
└─────────────────────────────────────────┘

关键理解点

  1. 双层 DNS 劫持：即使指定 DNS 服务器也无法绕过
  2. Fake-IP 是桥梁：连接 DNS 劫持和流量处理
  3. 映射表是核心：实现基于域名的规则匹配
  4. 策略路由是基础：内核级流量拦截
  5. 完全透明：应用程序无感知

🎯 常见问题

Q: 为什么指定 8.8.8.8 仍返回 Fake-IP？ A: 策略路由在内核级别拦截了所有流量，Clash 检测到端口 53 的流量就拦截并返回 Fake-IP。

Q: Fake-IP 会变吗？ A: 短期内不会。Clash 缓存映射关系（几小时到几天），同一域名返回相同的 Fake-IP。

Q: 直连流量也走 TUN 吗？ A: 是的。所有流量都进入 Clash，由 Clash 根据规则决定代理还是直连。

Q: 如何区分代理和直连？ A: Clash 通过反查映射表得到域名，然后匹配规则列表。

Q: 为什么不用 iptables？ A: Fake-IP 模式更优雅：DNS 阶段就标记流量，策略路由自动捕获，无需复杂的 iptables 规则。

🔚 总结

Clash TUN 模式是一个完整的网络栈中间层：

  • 对上：伪装成 DNS 服务器，返回 Fake-IP
  • 对中：维护映射表，连接 DNS 和流量处理
  • 对下：接管 TUN 流量，智能分流

核心是 Fake-IP 映射表：

  • DNS 服务写入映射
  • 流量处理读取映射
  • 实现基于域名的规则匹配

这就是透明代理的完整工作原理！🎓
